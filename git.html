Git 101

Some common programming issues

  • We need to regularly back up our code
  • We need to keep "milestones" when our code actually works!
  • We need to keep a history of our codebase
  • We need to share our code with other developers
  • We need to experiment with our code without affecting the main "branch"

Enter Version Control

There are two main types of version control.

Centralized version control

  • You have to check-out the files you need to edit
  • The server's administrators have complete control over the codebase - not ideal for open source projects
  • Coders are tied to the network to make changes

Distributed version control

  • Every developer has their own copy of the code, and all copies are seen as equally "valid"
  • There may be a "central" repository, but developers can work anywhere with their local copies
  • Git encourages small, incremental changes, and makes merging code between copies easy

Some Jargon

  • Repository - a collection of code
  • Staged - when we have told Git that we want to save our changes
  • Committed - when we have saved our changes to Git
  • Reverted - when we roll-back changes that have been saved
  • Merge - resolving differences between files

Git workflow

  • Initialize Git on a directory - this is our repository, or "repo"
  • Add a file, and "Stage" it
  • Commit the file, so that it exists in Git's history
  • Edit the file, Stage the changes, and Commit
  • Git logs the changes and considers the files to be unmodified again.
